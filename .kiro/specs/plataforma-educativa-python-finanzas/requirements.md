# Documento de Requisitos

## Introducción

La Plataforma Educativa de Python para Análisis Financiero es un sistema interactivo altamente didáctico diseñado para el curso "Lenguajes de Programación" de Ingeniería Financiera. La plataforma ofrece una experiencia de aprendizaje moderna e interactiva inspirada en plataformas como Codecademy y DataCamp, especializada en análisis financiero con énfasis en estudiantes que nunca han programado.

Los estudiantes aprenden Python mediante tutoriales guiados paso a paso con explicaciones visuales animadas, retroalimentación en tiempo real, sistema de pistas progresivas, y asistente de depuración visual. La plataforma incluye visualizaciones interactivas que responden a cambios en el código, calculadoras financieras manipulables, biblioteca de snippets de código financiero, y playground de experimentación libre.

Una característica distintiva es la comparación lado a lado de código Python y RStudio, mostrando equivalencias entre yfinance/pandas/matplotlib en Python y quantmod/tidyverse/ggplot2 en R, permitiendo a estudiantes con conocimientos de R entender las similitudes y diferencias conceptuales entre ambos lenguajes.

La gamificación está integrada mediante insignias, puntos de experiencia, barras de progreso, desafíos diarios, mini-desafíos de refuerzo, y tabla de clasificación anónima. Los estudiantes reciben mensajes motivacionales, animaciones de celebración, y feedback visual constante. La interfaz incluye quizzes interactivos embebidos, videos tutoriales integrados, navegador de documentación, y transiciones animadas suaves.

Los profesores pueden monitorear el progreso detallado, revisar ejercicios, gestionar entregas de proyectos en equipo, y exportar reportes. El sistema se utiliza durante clases en vivo compartidas por Microsoft Teams, con integración a GitHub para control de versiones y trabajo colaborativo.

## Requisitos

### Requisito 1: Autenticación Institucional

**Historia de Usuario:** Como administrador del sistema, quiero que solo estudiantes con correos institucionales puedan registrarse, para garantizar que solo usuarios autorizados accedan a la plataforma.

#### Criterios de Aceptación

1. WHEN un usuario intenta registrarse, THE Sistema SHALL validar que el correo electrónico termine en @correo.itm.edu.co
2. IF un usuario intenta registrarse con un correo que no es Correo_Institucional, THEN THE Sistema SHALL rechazar el registro y mostrar un mensaje de error
3. WHEN un usuario completa el formulario de registro con un Correo_Institucional válido, THE Sistema SHALL generar un Código_Verificación de 6 dígitos
4. WHEN el Sistema genera un Código_Verificación, THE Sistema SHALL enviar el código al Correo_Institucional del usuario
5. WHEN un usuario ingresa el Código_Verificación correcto dentro de 15 minutos, THE Sistema SHALL activar la cuenta del usuario
6. IF un usuario ingresa un Código_Verificación incorrecto, THEN THE Sistema SHALL mostrar un mensaje de error y permitir reintentos hasta 3 veces
7. IF un Código_Verificación no es utilizado dentro de 15 minutos, THEN THE Sistema SHALL invalidar el código y requerir generación de uno nuevo

### Requisito 2: Gestión de Lista Autorizada y Sistema de Invitaciones

**Historia de Usuario:** Como Profesor, quiero cargar un archivo CSV con correos autorizados para un semestre específico y enviar invitaciones, para controlar qué estudiantes pueden registrarse en cada periodo académico.

#### Criterios de Aceptación

1. THE Sistema SHALL permitir al Profesor cargar un archivo CSV con Lista_Autorizada asociado a un Semestre específico
2. WHEN el Profesor carga un archivo CSV, THE Sistema SHALL validar que cada línea contenga un Correo_Institucional válido
3. IF el archivo CSV contiene correos inválidos, THEN THE Sistema SHALL mostrar un reporte de errores con las líneas incorrectas
4. WHEN el Profesor carga un archivo CSV válido, THE Sistema SHALL generar una Invitación_Semestre única para cada Correo_Institucional
5. WHEN el Sistema genera invitaciones, THE Sistema SHALL enviar un correo electrónico a cada Correo_Institucional con el enlace de Invitación_Semestre
6. WHEN un Estudiante accede a una Invitación_Semestre válida, THE Sistema SHALL permitir el registro sin requerir Código_Verificación
7. THE Sistema SHALL validar que cada Invitación_Semestre sea válida únicamente para el Semestre asociado
8. THE Sistema SHALL permitir al Profesor ver el Estado_Invitación de cada estudiante invitado (Enviada, Registrada, Pendiente)
9. THE Sistema SHALL permitir al Profesor reenviar invitaciones a estudiantes con Estado_Invitación Pendiente
10. WHEN un Estudiante completa el registro mediante Invitación_Semestre, THE Sistema SHALL actualizar el Estado_Invitación a Registrada
11. IF un Correo_Institucional no tiene una Invitación_Semestre válida, THEN THE Sistema SHALL rechazar el registro con un mensaje explicativo

### Requisito 3: Roles de Usuario

**Historia de Usuario:** Como usuario del sistema, quiero tener permisos específicos según mi rol, para acceder solo a las funcionalidades que me corresponden.

#### Criterios de Aceptación

1. THE Sistema SHALL asignar el rol de Estudiante o Profesor a cada usuario registrado
2. WHEN un Profesor inicia sesión, THE Sistema SHALL mostrar funcionalidades de monitoreo, revisión y gestión
3. WHEN un Estudiante inicia sesión, THE Sistema SHALL mostrar funcionalidades de aprendizaje, ejercicios y entregas
4. THE Sistema SHALL restringir el acceso a funcionalidades según el rol del usuario autenticado

### Requisito 4: Interfaz de Laboratorio

**Historia de Usuario:** Como Estudiante, quiero una interfaz tipo laboratorio con múltiples paneles, para seguir la clase, escribir código y ver resultados simultáneamente.

#### Criterios de Aceptación

1. THE Sistema SHALL mostrar cuatro paneles en la interfaz: Panel_Contenido, Panel_Código, Panel_Resultados y Panel_Ejercicios
2. THE Panel_Contenido SHALL mostrar explicaciones teóricas sobre variables, listas, diccionarios y análisis financiero
3. THE Panel_Código SHALL proporcionar un editor interactivo de Python con resaltado de sintaxis
4. THE Panel_Resultados SHALL mostrar la salida de ejecución del código incluyendo tablas, gráficos y métricas financieras
5. THE Sistema SHALL permitir al usuario redimensionar los paneles según sus preferencias
6. THE Sistema SHALL mantener el estado de los paneles durante la sesión del usuario

### Requisito 5: Ejecución de Código Python

**Historia de Usuario:** Como Estudiante, quiero ejecutar código Python en el navegador, para practicar programación sin instalar software adicional.

#### Criterios de Aceptación

1. WHEN un Estudiante escribe código Python en el Panel_Código, THE Sistema SHALL permitir ejecutar el código mediante un botón o atajo de teclado
2. WHEN el código se ejecuta exitosamente, THE Sistema SHALL mostrar la salida en el Panel_Resultados dentro de 5 segundos
3. IF el código contiene errores de sintaxis o ejecución, THEN THE Sistema SHALL mostrar el mensaje de error con número de línea en el Panel_Resultados
4. THE Sistema SHALL limitar el tiempo de ejecución de código a 30 segundos por ejecución
5. IF el código excede 30 segundos de ejecución, THEN THE Sistema SHALL detener la ejecución y mostrar un mensaje de timeout
6. THE Sistema SHALL proporcionar acceso a bibliotecas estándar de Python y bibliotecas financieras como yfinance, pandas, matplotlib y numpy

### Requisito 6: Ejercicios Interactivos

**Historia de Usuario:** Como Estudiante, quiero resolver ejercicios interactivos con validación automática, para reforzar mi aprendizaje y recibir retroalimentación inmediata.

#### Criterios de Aceptación

1. THE Sistema SHALL mostrar Ejercicios organizados por tema en el Panel_Ejercicios
2. WHEN un Estudiante selecciona un Ejercicio, THE Sistema SHALL mostrar el enunciado y casos de prueba esperados
3. WHEN un Estudiante ejecuta su solución, THE Sistema SHALL validar automáticamente si la solución es correcta
4. IF la solución es correcta, THEN THE Sistema SHALL marcar el Ejercicio como completado y actualizar el Progreso_Estudiante
5. IF la solución es incorrecta, THEN THE Sistema SHALL mostrar qué casos de prueba fallaron sin revelar la solución completa
6. THE Sistema SHALL permitir al Estudiante reintentar un Ejercicio múltiples veces
7. THE Sistema SHALL registrar la fecha y hora de completación de cada Ejercicio

### Requisito 7: Evaluaciones Tipo ECAES

**Historia de Usuario:** Como Profesor, quiero crear evaluaciones con preguntas tipo ECAES, para medir el conocimiento teórico de los estudiantes.

#### Criterios de Aceptación

1. THE Sistema SHALL permitir al Profesor crear Evaluaciones con preguntas de opción múltiple
2. WHEN un Estudiante completa una Evaluación, THE Sistema SHALL calcular y mostrar la calificación automáticamente
3. THE Sistema SHALL registrar las respuestas y calificación de cada Evaluación en el Progreso_Estudiante
4. WHEN el Profesor revisa una Evaluación, THE Sistema SHALL mostrar las respuestas de cada Estudiante y la calificación obtenida
5. THE Sistema SHALL permitir al Profesor establecer una fecha límite para cada Evaluación
6. IF un Estudiante intenta acceder a una Evaluación después de la fecha límite, THEN THE Sistema SHALL mostrar un mensaje indicando que la Evaluación está cerrada

### Requisito 8: Integración con GitHub

**Historia de Usuario:** Como Estudiante, quiero conectar mi cuenta de GitHub, para vincular repositorios y trabajar en proyectos con control de versiones.

#### Criterios de Aceptación

1. THE Sistema SHALL permitir al Estudiante conectar su cuenta de GitHub mediante OAuth
2. WHEN un Estudiante conecta su cuenta de GitHub, THE Sistema SHALL almacenar el token de acceso de forma segura
3. THE Sistema SHALL permitir al Estudiante vincular un repositorio de GitHub a su perfil
4. WHEN un Estudiante vincula un repositorio, THE Sistema SHALL verificar que el Estudiante tenga permisos de acceso al repositorio
5. IF el Estudiante no tiene permisos de acceso, THEN THE Sistema SHALL mostrar un mensaje de error y no vincular el repositorio
6. THE Sistema SHALL permitir al Estudiante desvincular su cuenta de GitHub en cualquier momento

### Requisito 9: Gestión de Equipos

**Historia de Usuario:** Como Estudiante, quiero crear equipos y trabajar colaborativamente, para desarrollar proyectos en grupo con mis compañeros.

#### Criterios de Aceptación

1. THE Sistema SHALL permitir a un Estudiante crear un Equipo con un nombre único
2. WHEN un Estudiante crea un Equipo, THE Sistema SHALL asignar al Estudiante como líder del Equipo
3. THE Sistema SHALL permitir al líder del Equipo invitar a otros Estudiantes mediante su Correo_Institucional
4. WHEN un Estudiante recibe una invitación, THE Sistema SHALL notificar al Estudiante y permitir aceptar o rechazar la invitación
5. WHEN un Estudiante acepta una invitación, THE Sistema SHALL agregar al Estudiante al Equipo
6. THE Sistema SHALL limitar el tamaño de cada Equipo a un mínimo de 2 y un máximo de 4 Estudiantes
7. IF un Equipo intenta agregar un quinto miembro, THEN THE Sistema SHALL rechazar la invitación y mostrar un mensaje de límite alcanzado
8. THE Sistema SHALL permitir al líder del Equipo vincular un Repositorio_Equipo de GitHub compartido
9. THE Sistema SHALL mostrar a todos los miembros del Equipo el Repositorio_Equipo vinculado

### Requisito 10: Entregas de Proyecto y Estructura de Evaluación

**Historia de Usuario:** Como Estudiante, quiero enviar entregas de proyecto según la estructura de evaluación del curso, para cumplir con los requisitos académicos y obtener mi calificación final.

#### Criterios de Aceptación

1. THE Sistema SHALL implementar la Estructura_Evaluación estándar con cinco Componentes_Evaluación: Trabajo_1 (20%), Trabajo_2 (20%), Concurso_Análisis_Financiero (20%), Examen_Final (20%) y Seguimiento (20%)
2. THE Sistema SHALL calcular automáticamente las Fechas_Entrega basadas en la configuración del Semestre y la duración total en semanas
3. WHEN el Profesor crea un Semestre, THE Sistema SHALL calcular Fecha_Entrega de Trabajo_1 en la semana 6 del Semestre
4. WHEN el Profesor crea un Semestre, THE Sistema SHALL calcular Fecha_Entrega de Trabajo_2 en la semana 11 del Semestre
5. WHEN el Profesor crea un Semestre, THE Sistema SHALL calcular Fecha_Entrega de Concurso_Análisis_Financiero en la semana 15 del Semestre
6. WHEN el Profesor crea un Semestre, THE Sistema SHALL calcular Fecha_Entrega de Examen_Final en la semana 17 del Semestre
7. WHEN se acerca una Fecha_Entrega, THE Sistema SHALL notificar a los Equipos con 7 días y 1 día de anticipación
8. THE Sistema SHALL asociar Trabajo_1 con contenido de Módulos 1 y 2, Trabajo_2 con contenido de Módulos 4 a 6, y Concurso_Análisis_Financiero con contenido de Módulos 7 y 8
9. THE Sistema SHALL permitir a un Equipo subir Trabajo_1, Trabajo_2 y Concurso_Análisis_Financiero como Entrega_Equipo que incluya notebook de Python, análisis financiero y visualizaciones
10. WHEN un miembro del Equipo sube una Entrega_Equipo, THE Sistema SHALL requerir confirmación de todos los miembros del Equipo
11. WHEN todos los miembros del Equipo confirman la Entrega_Equipo, THE Sistema SHALL marcar la entrega como válida y registrar la fecha de submisión
12. IF no todos los miembros confirman antes de la Fecha_Entrega, THEN THE Sistema SHALL marcar la Entrega_Equipo como no válida
13. THE Sistema SHALL permitir al Profesor revisar todas las entregas y agregar comentarios de retroalimentación con calificación numérica
14. IF un Equipo intenta subir una Entrega_Equipo después de la Fecha_Entrega, THEN THE Sistema SHALL marcar la entrega como tardía
15. THE Sistema SHALL gestionar el Examen_Final como Evaluación_Individual donde cada Estudiante completa la evaluación de forma independiente
16. THE Sistema SHALL calcular automáticamente el Seguimiento (20%) basado en ejercicios completados, actividad de código, participación en clase y progreso a través de módulos
17. THE Sistema SHALL mostrar al Estudiante un dashboard con el desglose de su calificación en los cinco Componentes_Evaluación
18. THE Sistema SHALL calcular la calificación final como la suma ponderada de los cinco Componentes_Evaluación
19. THE Sistema SHALL permitir al Profesor exportar las calificaciones de todos los Componentes_Evaluación en formato CSV

**Consideraciones Técnicas Importantes:**

1. **Gestión de Semestres**: El Requisito 33 introduce la capacidad de reutilizar la plataforma semestre tras semestre. La estructura curricular (8 módulos) permanece constante, pero las fechas y estudiantes cambian por semestre. Esto requiere:
   - Asociación de todas las entidades (estudiantes, equipos, entregas) a un semestre específico
   - Cálculo automático de fechas de entrega basado en la duración del semestre
   - Capacidad de archivar semestres antiguos manteniendo datos históricos

2. **Sistema de Invitaciones**: El Requisito 34 reemplaza el simple CSV upload con un sistema de invitaciones por correo electrónico. Cada invitación es única y está asociada a un semestre específico, simplificando el proceso de registro.

3. **Calificaciones Estandarizadas**: El Requisito 35 formaliza el sistema de calificaciones con porcentajes fijos (5 × 20% = 100%). El componente de Seguimiento se calcula automáticamente basado en la actividad del estudiante en la plataforma.

4. **Integración con Google Colab**: El Requisito 37 proporciona flexibilidad a los estudiantes para trabajar en un entorno profesional de notebooks. La integración bidireccional (exportar/importar) permite:
   - Estudiantes pueden abrir su código en Google Colab con un clic
   - Trabajo realizado en Colab puede sincronizarse de vuelta a la plataforma
   - Entregas pueden enviarse directamente desde notebooks de Colab
   - Parser de .ipynb debe implementar propiedad round-trip para garantizar integridad

5. **Comparación Python/R**: La generación de Código_Equivalente_R debe mapear correctamente las bibliotecas:
   - yfinance (Python) → quantmod (R)
   - pandas (Python) → tidyverse/dplyr (R)
   - matplotlib/seaborn (Python) → ggplot2 (R)
   - numpy (Python) → base R

6. **Interactividad**: Los requisitos 17-32 transforman la plataforma en un entorno altamente interactivo y didáctico, esencial para estudiantes sin experiencia previa en programación.

7. **Parsers Críticos**: Los requisitos 15, 16 y 37 definen parsers con propiedades round-trip obligatorias para garantizar integridad de datos. El parser de .ipynb debe manejar:
   - Conversión bidireccional entre código Python y formato notebook
   - Preservación de metadata y estructura de celdas
   - Validación de sintaxis en importación

8. **Estructura de Evaluación**: Los Requisitos 10 y 35 implementan la estructura estándar de evaluación (5 componentes × 20% = 100%) que debe mantenerse consistente cada semestre, con fechas calculadas automáticamente según el Requisito 33. Trabajo 1 cubre Módulos 1-2, Trabajo 2 cubre Módulos 4-6, y Concurso cubre Módulos 7-8.

9. **Currículo Flexible**: El Requisito 36 define una estructura curricular de 8 módulos que se adapta a la duración del semestre (típicamente 17 semanas) con entregas alineadas a la estructura de evaluación.

10. **Entrega Incremental de Contenido**: El MVP entrega los primeros 3 módulos completos, permitiendo que los estudiantes comiencen el curso y completen Trabajo 1. Las fases subsecuentes agregan 2 módulos por fase, alineados con las entregas evaluativas.

Todos los requisitos siguen los patrones EARS y las reglas de calidad INCOSE para garantizar claridad, precisión y verificabilidad.
